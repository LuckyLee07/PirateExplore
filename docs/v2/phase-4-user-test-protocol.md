# V2 阶段 4：目标用户测试协议

版本：1.0

适用构建：V2 第一章样片，`CURRENT_PHASE = 4`

状态：协议已就绪，真实外部测试结果尚未采集

## 1. 测试目的

本测试只回答是否值得扩建前三海域，不用于证明所有玩家都喜欢产品。需要验证四个立项问题：

1. 玩家能否在不接受口头教学的情况下完成第一次出航；
2. 玩家能否说明舰炮阶段怎样影响接舷阶段；
3. 完成返航与升级后，玩家是否主动想开始第二次远航；
4. 玩家首先记住的是海图、船战、符文或船员，而不是菜单和资源表。

自动测试、开发者试玩和模拟器截图不能替代这些结论。

## 2. 样本与轮次

每轮建议 5 名、合计至少 10 名有效外部目标用户。参与者不得是本项目开发者、策划案作者或已经看过完整流程的人。

- 核心样本：18 岁以上，过去一年玩过至少一款移动端 RPG、策略或冒险游戏；
- 经验分层：每轮至少 2 名轻度玩家、2 名中重度玩家；
- 设备分层：每轮至少覆盖一台较小屏幕 iPhone 和一台常规/大屏 iPhone；
- 无效场次：安装失败、系统来电等外因导致无法继续的场次记为技术失败，不计入理解率分母，但问题仍进入登记表；
- 不得用团队成员的“模拟用户回答”补足样本。

第一轮是诊断轮：寻找 P0/P1、确定玩家真实理解路径。修复后必须重新构建。第二轮是验证轮：使用未参加第一轮的新用户，按质量门槛做 Go/No-Go 判断。

## 3. 隐私与测试记录

测试前告知参与者：本构建只在本地保存操作事件，不上传网络；记录内容包括页面阶段、选择、资源与战斗数值，不记录姓名、账号、联系人、定位或录音。参与者编号使用 `R1-P01` 这类匿名标识。

若需要录屏、录音或拍摄手部操作，必须另行取得明确同意；拒绝录制不影响参与。测试结束后将人工观察填入 [`phase-4-test-record-template.csv`](phase-4-test-record-template.csv)，本地事件只用于核对时序，不替代观察者记录。

## 4. 测试环境与构建冻结

每轮开始前记录：Git 提交、构建日期、设备、iOS 版本、横竖屏、静音键状态和是否首次安装。所有参与者必须使用同一轮冻结构建和全新 `player` 测试档。

开始前执行：

1. 运行 `tools/v2/validate_phase4.sh`；
2. 完成 arm64 模拟器和免签名 device compile；
3. 删除该设备上此前的测试档，确认从“瓶中召唤”开始；
4. 关闭通知，保持网络条件一致；V2 本身不依赖网络；
5. 准备计时器、观察表和问题登记表；
6. 不在参与者面前展示策划案、流程图或测试门槛。

## 5. 主持人开场脚本

只朗读以下信息，不解释玩法：

> 这是一个仍在开发中的海盗冒险样片。请像第一次下载游戏一样操作，想到什么可以说出来。这里测试的是产品，不是你；遇到困惑可以直接说，但我不会替你做选择。请玩到你认为这一段已经结束，或者确实无法继续为止。

允许回答设备和测试范围问题；不允许解释按钮含义、最优路线、资源用途、舰炮目标或胜负原因。

## 6. 无引导任务流程

主持人只给出一个任务：“开始游戏并继续玩到你认为这一章结束。”之后静默观察。

必须记录以下时间点：

- `T0`：画面可操作；
- `T1`：玩家能复述当前目标；
- `T2`：第一次点击“驶入第一片迷雾”；
- `T3`：进入第一次舰炮战；
- `T4`：进入接舷战；
- `T5`：取得符文线索；
- `T6`：返航并完成一次升级；
- `T7`：玩家主动点击、寻找或询问第二次远航。

同时记录：每次停顿超过 10 秒、反复点击、误触、返回尝试、需要主持人干预的位置、失败原因和恢复选择。只有玩家连续 90 秒无进展、明确表示无法继续或遇到 P0 时，主持人才可结束任务并记录阻断。

## 7. 结束访谈

完成或结束任务后，按顺序提问，不纠正答案，不使用“甲板传递”“符文目标”等提示词：

1. 你认为刚才的目标是什么？
2. 你为什么选那条路线？
3. 前面的舰炮战对后面的战斗产生了什么影响？
4. 你得到的战利品有什么用？
5. 如果继续玩，你下一步会做什么？
6. 哪个时刻最像海盗冒险？
7. 哪个时刻最像在操作菜单？
8. 你最想继续了解哪个角色、海域或符文？
9. 如果这是一款完整产品，你倾向于一次购买、首章试玩后一次解锁，还是其他方式？为什么？

最后才可以向参与者解释测试目的。

## 8. 指标判定规则

所有比例以第二轮有效参与者为最终分母；第一轮单独报告，不与第二轮合并稀释问题。

| 指标 | 单人通过定义 | 第二轮门槛 |
| --- | --- | --- |
| 独立完成首次远航 | `T2` 前无主持人玩法提示，且无阻断 | ≥ 70% |
| 理解双阶段战斗 | 能明确说出攻击甲板会削弱接舷敌人；只说“先打船再打人”不通过 | ≥ 70% |
| 第二次出航意愿 | `T6` 后主动寻找/询问下一航程，或访谈明确表示立刻继续 | ≥ 60% |
| 核心幻想记忆 | 对“刚才是什么游戏”的第一组描述包含海图、船战、符文或具体船员 | ≥ 70% |

时间、失败次数、路线偏好和按钮误触用于诊断，不单独决定立项。若一名玩家同时遇到产品 P0 和外部技术失败，以 P0 计入产品结果。

## 9. 问题分级与修复闭环

- P0：崩溃、存档损坏、首章死路、关键操作完全不可执行；立即停止该构建测试；
- P1：目标不清、关键操作无法理解、胜负原因无法解释；第二轮前必须清零；
- P2：节奏、表现、重复、音效或帧率明显影响体验；明确负责人和是否阻断立项；
- P3：文案、局部视觉和非阻断润色；进入正常 backlog。

每个问题必须包含匿名参与者编号、复现步骤、期望/实际结果、证据、严重度、负责人、目标构建和复验结果。修复后由未实现该修复的人复验。

## 10. 两轮执行节奏

### 第一轮：诊断

逐人完成测试并每日归并问题。5 人全部结束后先处理 P0/P1，再根据频率和影响选择 P2；不得在同一轮中途更换构建后混算数据。输出第一轮原始记录、问题登记和改动清单。

### 第二轮：验证

使用新构建和 5 名新参与者重复完全相同的协议。冻结数据后计算四项比例，逐项对照 `quality_gate.csv`。输出第二轮原始记录、指标汇总、遗留问题和最终决策。

## 11. 放行规则

只有同时满足以下条件才建议扩建：第二轮四项外测门槛全部通过；P0/P1 为零；性能、存档、资源和构建内部门槛通过；第二张地图预算不超过 8 人周。

任何一项未通过都返回第一章纵切迭代，不得以平均分、团队主观认可或内部试玩替代。当前状态为“待外测”，不是 Go。
