## 1. 执行摘要

### 1.1 新的产品定位

NewPirate 不应继续以“轻量海盗经营 RPG”为中心，而应重新定位为：

> **剧情驱动的暗黑奇幻海盗探索 RPG：玩家被吸入受诅咒的瓶中世界，
> 整备船只与船员，探索迷雾海域，经历舰炮战与接舷战，收集十六枚
> 封印符文，并逐步揭开海盗王复活阴谋。**

经营、生产、建造继续存在，但定位从“游戏主体”调整为“远航准备、
队伍成长与港口反馈”。

### 1.2 产品核心价值

项目最有辨识度的四项内容是：

1. 水晶瓶、掠夺者号、海盗王诅咒和十六符文的主线；
2. 羊皮纸海图、战争迷雾、据点与航线选择；
3. 舰炮战自然转入接舷战的双阶段战斗；
4. 船员职业与船只升级共同影响一次远航。

后续所有开发都应直接强化至少一项核心价值。无法强化核心价值、只会
增加系统数量的需求，默认后置。

### 1.3 当前项目判断

当前工程不是空白原型，而是旧完整版本的复原工程。探索、剧情、随机
事件、船战、接舷战、成就、排行、永恒竞技场、商店和充值等模块均已
有代码与配置基础。

因此主要矛盾不是“功能太少”，而是：

- 旧系统过多，主角不清晰；
- 剧情、探索与战斗被生产列表和付费入口遮蔽；
- 内容表完成度不一致，版本之间存在冲突；
- 美术气质统一，但界面构图和移动端可读性已经过时；
- 数值、内容生产和测试缺少可持续的工程化链路。

### 1.4 第一目标

第一阶段不制作十六海域的完整游戏，只制作一个 15～20 分钟的首章纵切：

> 玩家打开水晶瓶进入瓶中世界，在皇家港完成一次最小整备，驶入迷雾
> 海域，经历事件、舰炮战和接舷战，取得第一枚符文线索，回港完成一次
> 有意义的升级，并感受到海盗王诅咒的存在。

这段体验必须在不依赖旧长期系统和强付费的情况下成立。

## 2. 原项目内容还原

### 2.1 世界观与主线

原始 V2 剧情的基本结构如下：

1. 邪恶海盗王及其船只“掠夺者号”被海神封印在带有十六枚符文的
   水晶瓶中；
2. 火山灾难导致符文散落，封印逐渐失效；
3. 玩家打开水晶瓶并被吸入瓶中世界；
4. 海盗王亡魂试图引导、腐化并最终占据玩家身体；
5. 玩家穿越十六片主题海域，接触海妖、黑暗精灵、龙骑士、海神及其他
   势力；
6. 玩家通过据点、首领和剧情节点收集符文，逐渐理解封印、海盗王与自身
   的关系；
7. 最终进入围绕复活、封印和玩家选择展开的冲突。

剧情资源的价值在于，它能把探索、战斗和成长统一成一场持续推进的冒险，
而不是十六张互不相关的刷图地图。

当前需要解决的问题：

- V1、V2、剧情 Excel 与运行时 `plot.csv` 并非同一版本；
- 部分角色和阵营只出现在文字梗概中，没有可玩的任务链；
- 剧情节点与地图编号绑定较深，但玩家动机和阶段目标表达不足；
- 原剧情适合做“章节骨架”，不适合直接作为游戏对白使用。

### 2.2 世界与地图结构

原设计包含十六张逐步扩大的地图：

- 海面格子可以移动，陆地和阻挡区不可移动；
- 玩家移动消耗体力，并逐格揭开战争迷雾；
- 地图包含出生点、传送、复活、酒馆、黑市、竞技场、资源据点、怪物
  据点、宝箱和随机事件；
- 地图探索达到一定比例后揭示出口；
- 后续地图使用地图碎片、钥匙和特殊钥匙控制进入；
- 每位玩家的部分据点可以随机生成，但生成后保持固定。

这套结构的优点是能制造“未知海域”和航线规划感；问题是原设计把地图
做得过大、依赖格子消耗和随机遇敌，容易变成重复移动。

新版本应保留“迷雾、航线和据点”，弱化“为了消耗体力而逐格点击”。

### 2.3 原始核心循环

原项目的完整循环为：

```text
生产与采集
  → 建造和制造
  → 招募、训练船员
  → 配置补给、背包与队伍
  → 进入海域并揭开迷雾
  → 触发事件、据点或敌人
  → 舰炮战
  → 接舷战
  → 选择和携带战利品
  → 返回港口
  → 强化生产、船只和船员
  → 解锁下一海域
```

循环覆盖面很广，但从新玩家视角看，前半段准备过长，真正体现海盗幻想
的探索与战斗出现太晚。

### 2.4 生产、建筑与资源

原生产体系包含多级金属、木材、皮革、布料、粉尘、鳞片、钥匙、背包、
工具和装备，并配套仓库、住宅、农场、商店、马厩、磨坊、酒馆、铁匠铺、
训练设施、矿区等建筑。

从历史表格可以看出，这套体系大量继承了传统地下城经营 RPG 的生产树，
部分职业和材料只是替换为海盗名称。它提供了足够多的成长节点，却没有
持续服务“航海、船只、船员和诅咒”主题。

结论：保留生产作为准备层，但必须压缩资源数量和建筑层级。

### 2.5 船员与职业

原职业树包含一至六星、多条转职分支、特殊招募角色和怪物单位，实际数量
远超首个可玩版本的需求。

这套结构的可复用价值是：

- 不同职业可分别影响舰炮战、接舷战、探索和续航；
- 酒馆招募天然适合承载人物故事；
- 职业选择可以形成远航前的策略取舍。

主要问题是职业主题不统一，部分职业仍是战士、刺客、牧师、法师等传统
奇幻职业的海盗换皮。

### 2.6 战斗结构

原战斗由两个阶段组成：

#### 舰炮战

- 双方按照回合自动开炮；
- 火炮数量和船只属性影响伤害；
- 随着回合推进，进入接舷战的概率提升；
- 部分技能、补给和船员可影响舰炮阶段。

#### 接舷战

- 双方最多各配置六名单位；
- 单位通过蓄力或冷却释放技能；
- 存在治疗、眩晕、持续伤害、攻击降低等状态；
- 玩家可使用有限补给进行恢复。

“先舰炮、后接舷”是值得重点重做的核心机制。原版本的问题主要在表现和
决策密度：舰炮阶段偏自动数值播放，接舷阶段信息拥挤，两个阶段的因果
联系不够直观。

### 2.7 随机事件与任务

原事件框架包括赌博、借贷资源、学习天赋、购买地图、出售钥匙等类型；
后续方案又把部分弹窗事件改成有倒计时的“情报任务”。

当前资料反映出两个问题：

- 事件数量少，容易快速重复；
- 任务表包含大量占位字段，不能直接作为正式内容库。

新版本应把事件变成“航线选择的后果”，优先制作有角色、有风险、有后续
回响的小型事件链，而不是继续增加纯资源兑换弹窗。

### 2.8 长期系统与商业化

原项目设计了：

- 350 层永恒竞技场；
- 探索与竞技场排行榜；
- 成就和七日登录；
- 钻石商店、充值档位和循环折扣礼包；
- 付费地图碎片、快速挑战、秒杀、任务立即完成；
- 战斗后宝箱踩中骷髅后付费继续。

这些系统说明旧版本按 2015 年移动免费游戏思路搭建，付费点几乎进入了
地图、战斗、任务和成长的每一层。

新版本不能默认继承这套商业化。产品在证明核心循环和留存前，应关闭或
隔离强付费入口。

## 3. 当前工程与资产基线

### 3.1 已实现的系统

现有 Lua 模块已经覆盖主要旧功能：

| 领域 | 主要模块 | 当前判断 |
|---|---|---|
| 主界面与数据 | `MainMenu.lua`、`DataManager.lua` | 可作为旧流程基线 |
| 生产与仓库 | `Resource.lua`、`BuildMode.lua`、`MakeMode.lua`、`Repository.lua` | 功能广，需压缩 |
| 远征准备 | `Expedition.lua`、`ExploreBagController.lua` | 可复用逻辑，需重构体验 |
| 海图探索 | `Explore.lua`、`FogManager.lua`、`WorldMapLayer.lua` | 核心复用区 |
| 事件与剧情 | `EventLayer.lua`、`EventManger.lua`、`PlotLayer.lua` | 框架可用，内容需重写 |
| 战斗 | `FightMode.lua`、`FightDataManager.lua` | 核心复用区，优先拆分 |
| 长期系统 | `Achievement.lua`、`Ranking.lua`、`EternalArenaController.lua` | 暂停扩展 |
| 商业化 | `DiamondStore.lua`、`ChargeMode.lua`、`ChargingView.lua` | 与首章解耦 |

### 3.2 数据现状

运行时已有地图、据点、船员、技能、剧情、任务、事件、天赋、商店、成就
和竞技场等 CSV 数据，但这些文件属于旧版本运行数据，不应继续作为策划
唯一源文件。

后续应建立：

```text
可读策划源表
  → 自动校验
  → 导出运行时数据
  → 构建时校验引用
  → 游戏加载
```

严禁长期同时人工维护 Excel、加密 CSV 和 Lua 常量三份互相独立的数据。

### 3.3 美术资产判断

值得保留和升级的资产：

- 羊皮纸世界地图和各海域地块；
- 战争迷雾、海岛、航线和据点符号；
- 深海、幽灵和诅咒类怪物的气质；
- 舰炮、爆炸、船只和技能粒子；
- 瓶子、符文、罗盘、绳索、船木、黄铜等主题元素。

不应直接延续的表现：

- 深青色界面框覆盖大多数屏幕；
- 小字号、密集列表和长日志占据主视觉；
- 所有功能使用近似的全屏菜单结构；
- 顶部多货币和底部多入口长期常驻；
- 只替换组件皮肤、不改变屏幕构图的批量换皮方案。

新的视觉原则是：

> **明亮、可探索的海上冒险作为常态；诅咒、亡灵和深海危险作为反差。**

## 4. 新产品定义

### 4.1 类型与一句话描述

类型：剧情驱动的海图探索 + 船队养成 + 双阶段战斗 RPG。

一句话描述：

> 整备船只和船员，驶入被诅咒的瓶中海域，在迷雾中选择航线、发现事件，
> 先以舰炮压制敌船，再登船决胜，夺回封印符文并揭开海盗王的阴谋。

### 4.2 目标体验

玩家在每次 5～10 分钟远航中应依次感受到：

1. **准备感**：我选择的船员和补给会影响这次出航；
2. **未知感**：迷雾后可能是资源、危险、角色或故事；
3. **取舍感**：继续深入、改变航线或带着战利品返航；
4. **战斗感**：舰炮和接舷是同一场海战的两个阶段；
5. **成长感**：战利品能清楚改善下一次远航；
6. **叙事感**：每一片海域都推动符文与诅咒主线。

### 4.3 产品支柱

#### 支柱 A：瓶中世界与十六符文

每片海域必须有独立主题、关键势力、符文线索和诅咒进展。主线不是背景
说明，而是解锁地图、首领和玩家能力的结构。

#### 支柱 B：海图探索与航线选择

地图的乐趣来自信息不完全和风险取舍，不来自重复点击格子。玩家应能根据
补给、船员特性和已知情报规划路线。

#### 支柱 C：舰炮战转接舷战

舰炮阶段创造优势，接舷阶段兑现优势。两个阶段必须共享船体损伤、士气、
状态或站位结果，让前一阶段的选择真正影响后一阶段。

#### 支柱 D：有身份的船员与船只

船员不是一串兵种数值。核心船员应有职业、技能、性格、事件倾向和剧情
关系；船只模块则决定玩家选择怎样的远航风格。

### 4.4 明确不做什么

首章阶段不做：

- 十六海域全量内容；
- 完整 77+ 职业或复杂六星转职树；
- 350 层竞技场和排行榜；
- 公会、PVP、多人协作；
- 大规模港口自由建造；
- 全套新 UI 皮肤批量替换；
- 新增礼包、抽卡或强制广告；
- 为迁移引擎而重写项目。

## 5. 目标核心循环

### 5.1 单次远航循环

```text
查看海图情报
  → 选择船员、船只模块和补给
  → 进入海域
  → 揭开迷雾并选择航线
  → 处理据点、事件或战斗
  → 决定继续深入或返航
  → 结算战利品与剧情
  → 升级船只、船员或港口功能
  → 获得下一次航行目标
```

目标时长：普通远航 5～10 分钟；首章教学远航 10～15 分钟。

### 5.2 首次 20 分钟体验

| 时间目标 | 玩家体验 | 必须传达的信息 |
|---|---|---|
| 0～1 分钟 | 打开水晶瓶、进入瓶中世界 | 我被困住了，海盗王正在苏醒 |
| 1～3 分钟 | 到达皇家港、获得基础船只与船员 | 我要整备并出海寻找符文 |
| 3～6 分钟 | 第一次进入迷雾海图 | 路线和补给会影响结果 |
| 6～10 分钟 | 事件或资源据点、遇到敌船 | 海图中有机会和风险 |
| 10～14 分钟 | 舰炮战转接舷战 | 两阶段属于同一场战斗 |
| 14～17 分钟 | 获得符文线索并返航 | 本次收获推进了主线 |
| 17～20 分钟 | 完成一次船只或船员升级 | 我知道下一次为什么要再出航 |

这些时间是体验目标，不是硬编码。实测时应记录玩家自然完成时间。

### 5.3 北极星行为

> 新玩家完成第一次远航，理解获得了什么，完成一次有效升级，并主动开始
> 准备第二次远航。

在没有正式埋点前，内部测试使用以下代理问题：

- 玩家 60 秒后能否说出当前目标；
- 玩家是否能在无开发者指导下出航；
- 玩家能否说明舰炮战与接舷战的关系；
- 玩家能否说明本次失败或胜利的主要原因；
- 玩家返航后是否知道至少一个有效升级方向；
- 玩家是否主动提出“下一片迷雾后有什么”。

## 6. 系统取舍矩阵

| 系统 | 决策 | 原因 | 首章处理 |
|---|---|---|---|
| 水晶瓶与十六符文主线 | 重点保留 | 独特世界观核心 | 完成开场与第一线索 |
| 海图、迷雾、据点 | 重点重做 | 主要探索乐趣 | 一张小型海图、三类据点 |
| 舰炮战与接舷战 | 重点重做 | 最强机制卖点 | 完整跑通一场双阶段战斗 |
| 船员 | 重做 | 可承载策略与角色故事 | 四类基础职业、四名角色 |
| 船只升级 | 重做 | 连接准备与战斗 | 一个船体升级、两个模块槽 |
| 港口 | 简化重做 | 需要成长反馈，不应成为列表大厅 | 四个功能入口 |
| 生产与制造 | 大幅压缩 | 原系统过深且主题弱 | 五种基础资源、少量配方 |
| 背包与战利品 | 简化 | 原手动搬运增加摩擦 | 容量取舍，不做逐件拖动 |
| 随机事件 | 内容重做 | 框架可用，内容不足 | 6～8 个首章事件 |
| 天赋 | 后置重做 | 原逻辑 intrusive、数值单薄 | 用船长特质原型代替 |
| 成就 | 后置 | 不影响核心验证 | 隐藏入口，保留代码 |
| 七日登录 | 归档 | 不能证明核心乐趣 | 不进入首章 |
| 排行榜 | 归档 | 当前无有效竞争生态 | 不进入首章 |
| 350 层竞技场 | 归档 | 内容和数值成本高 | 保留代码，不开放 |
| 黑市 | 后置 | 适合后续风险经济 | 首章只做剧情演示或不开放 |
| 钻石商城与充值 | 隔离 | 旧付费方式破坏信任 | 首章不主动展示 |
| 地图碎片付费补齐 | 移除 | 直接阻断主线探索 | 改为探索或任务获得 |
| 战斗秒杀 | 移除 | 跳过核心卖点 | 不保留 |
| 宝箱付费续开 | 移除 | 奖励阶段制造挫败 | 改为清晰战利品选择 |
| 付费立即完成任务 | 移除 | 消解事件和任务意义 | 不保留 |

## 7. 首章纵切范围

### 7.1 章节目标

首章暂定为“皇家港与瓶中海域”，目标是一次性验证：

- 故事钩子是否成立；
- 玩家是否愿意进入迷雾；
- 远航准备是否产生选择；
- 双阶段战斗是否容易理解；
- 战利品是否能驱动第二次出航。

### 7.2 内容清单

#### 剧情

- 水晶瓶开场；
- 海盗王第一次低语；
- 皇家港引导角色；
- 第一枚符文线索；
- 首次返航后的诅咒选择或伏笔。

#### 地图

- 一张 7～12 个有效节点的小型海图；
- 一条安全路线和一条高风险捷径；
- 一个资源据点；
- 一个角色/事件据点；
- 一个敌船据点；
- 一个章节终点或首领节点。

节点数量以“每一步都有信息或选择”为准，不追求面积。

#### 船员

- 炮手：强化舰炮或削弱敌舰；
- 水手：接舷前排与保护；
- 航海士：视野、航线或事件判断；
- 医师：治疗和远航续航。

每个职业只提供一个清晰主动能力和一个清晰被动影响。

#### 战斗

- 一场教学敌船；
- 一场完整双阶段战斗；
- 一种可观察的舰炮优势传递到接舷阶段；
- 一个首章敌人特性；
- 胜利、失败和撤退结果均可解释。

#### 成长

- 一次船体耐久升级；
- 两个船只模块中的一次选择；
- 一次船员技能或职业强化；
- 一条下一次出航目标。

### 7.3 四个英雄画面

首章优先打磨四个高价值画面：

1. **港口/整备**：船、船员、补给和出航目标集中呈现；
2. **羊皮纸海图**：船只、迷雾、航线、据点和风险可读；
3. **舰炮战/接舷战**：战斗主体占据屏幕，减少数值和边框干扰；
4. **返航结算**：战利品、符文进展和下一次升级建立直接关系。

其他列表页面沿用旧资源或暂时隐藏，不能抢占首章美术排期。

### 7.4 首章验收门槛

首章进入下一阶段前必须同时满足：

- 新存档可稳定完成开场、整备、探索、战斗、返航和升级；
- 无必须阅读外部说明才能继续的步骤；
- 无强制或主动弹出的充值入口；
- 测试者能复述当前目标、战斗结果和下次升级方向；
- iOS 模拟器与设备编译保持通过；
- 首章流程不存在缺图、缺表、死路和不可恢复存档；
- 四个英雄画面至少完成一轮真机截图评审。

## 8. 重点系统重做原则

### 8.1 港口与整备

港口应表现为“下一次出航前的决策空间”，而不是所有系统按钮的总菜单。

首章只保留四个功能区域：

- 码头：船只、模块与维修；
- 酒馆：招募和船员故事；
- 工坊：少量关键制造；
- 海图桌：目标、情报与出航。

整备界面应默认给出可出航配置，并允许玩家围绕风险进行调整。不能要求
玩家先理解背包、食物、人数、容量、职业和多个资源上限后才能点击出发。

### 8.2 海图探索

新探索强调：

- 可见目标与未知迷雾并存；
- 路线长度、风险和收益可以比较；
- 航海士、情报和船只模块改变信息质量；
- 事件选择影响后续节点或战斗；
- 玩家可以主动返航，而不是只能耗尽资源。

减少：

- 没有决策的逐格点击；
- 高频、无关联的随机战斗；
- 单纯依靠体力阻塞探索；
- 需要付费才能继续主线的地图门槛。

### 8.3 双阶段战斗

舰炮战必须提供至少一种玩家可理解的决策：目标部位、炮击时机、船员技能
或是否提前接舷。接舷战必须兑现舰炮结果，例如：

- 炮击破坏敌方甲板，使敌人开场受伤；
- 击毁桅杆，降低敌方接舷速度；
- 保留火药，换取接舷阶段技能资源；
- 船体严重受损时继续接舷存在沉船风险。

首章只选择其中一种传递关系做透，避免同时实现所有机制。

战斗反馈优先级：

1. 当前阶段和目标；
2. 伤害来源与关键状态；
3. 舰炮结果如何影响接舷；
4. 胜负原因；
5. 数值细节。

### 8.4 船员与船只

船员成长由“星级和长职业链”改为：

```text
角色身份
  + 职业功能
  + 一个主动技能
  + 一个远航特性
  + 少量关键升级
```

船只成长围绕模块形成风格：

- 加固船体：容错与长航；
- 重炮甲板：舰炮优势；
- 快帆与舵：路线和先手；
- 医疗舱或补给舱：接舷续航。

首章只实现两种互斥倾向，验证玩家是否理解选择。

### 8.5 资源与经济

首阶段建议压缩为五种基础资源：

| 资源 | 主要用途 | 设计要求 |
|---|---|---|
| 金币 | 招募、交易、通用升级 | 来源和用途清晰 |
| 木材 | 船体维修和升级 | 与受损状态直接关联 |
| 铁料 | 火炮、护甲和工具 | 支撑战斗成长 |
| 补给 | 远航距离和恢复 | 形成风险预算，不做纯等待门槛 |
| 符文尘 | 主线或特殊能力 | 稀有，不能成为泛用垃圾货币 |

首章每种资源最多服务两个主要用途。新增资源必须说明它创造了什么新决策，
不能只为延长生产链。

### 8.6 剧情与事件

剧情表达采用三层：

- 主线演出：开场、符文、首领和海盗王诅咒；
- 航行对话：船员对地图、事件和选择的短反馈；
- 事件链：一次选择在后续节点产生回响。

首章事件以 6～8 个为宜，至少包含：

- 资源与风险交换；
- 船员立场冲突；
- 诅咒诱惑；
- 环境危险；
- 救援或掠夺选择；
- 与第一符文线索相关的事件。

### 8.7 奖励与返航

返航结算必须回答：

1. 我获得了什么；
2. 它能改善什么；
3. 主线推进了多少；
4. 下一次出航有什么新目标。

取消战利品逐件拖入背包和踩骷髅后付费续开的旧结算。容量选择应发生在
探索过程中，并在返航时自动形成清晰结果。

## 9. 美术与交互落地原则

### 9.1 视觉定位

长期视觉方向：

> **精致卡通 2.5D 海盗冒险 + 轻暗黑海怪与诅咒。**

常态画面使用明亮海蓝、阳光、羊皮纸、暖木和黄铜；危险和剧情高潮使用
紫色、冷青、深海黑和符文光。暗色用于制造对比，不再覆盖所有管理界面。

### 9.2 资产策略

资产处理顺序：

1. 先完成四个英雄画面的构图稿；
2. 再定义按钮、卡片、状态条等局部组件；
3. 在真机上验证字号、触控和主体占比；
4. 确认画面优于旧版后，才批量生产兼容资产；
5. 旧资产在新画面验收前不得批量覆盖。

优先复用地图、怪物和特效；UI 框体和页面布局按画面重新设计。

### 9.3 移动端可读性

- 主要行动按钮必须单手可触达；
- 战斗主体优先于装饰框和长期货币；
- 正文和数值必须在真机截图上可读；
- 同屏主要货币不超过三种；
- 首章界面不展示与当前目标无关的红点和入口；
- 支持安全区和不同纵向 iPhone 尺寸；
- 动画应提供反馈，不得延迟连续操作。

## 10. 工程与数据实施计划

### 10.1 近期技术路线

使用现有 Cocos2d-x/Lua 工程完成首章纵切，不在核心玩法验证前迁移引擎。

原因：

- 现有工程已经具备主循环的大部分功能；
- 引擎迁移不能解决定位、节奏、内容和战斗反馈问题；
- 当前最重要的是快速验证可玩体验，而不是追求技术栈现代化。

引擎迁移只在出现明确平台阻塞、维护成本无法接受或纵切验证成功且需要长期
扩张时重新评估。

### 10.2 代码治理优先级

P0：

- 保持 iOS 模拟器和设备构建稳定；
- 给首章提供独立、可重置的测试存档；
- 将首章外商城、礼包和长期系统入口隔离；
- 为数据加载、首章流程和战斗切换增加冒烟检查；
- 记录缺失资源、配置引用和 Lua 运行错误。

P1：

- 拆分 `FightMode.lua` 的战斗数据、阶段控制、UI 和奖励职责；
- 拆分 `Explore.lua` 的地图状态、移动、事件、迷雾和 UI 职责；
- 收敛 `DataManager.lua` 的全局数据职责；
- 建立统一的事件命名和生命周期管理；
- 对安全区、字号和触控范围建立共享适配层。

P2：

- 建立可重复的内容导出工具；
- 增加自动化截屏和关键流程回归；
- 清理失效 SDK、旧平台工程和不再使用的资源；
- 评估更长期的引擎与平台策略。

### 10.3 数据规范

每张正式表至少具备：

- 唯一 ID；
- 版本和负责人；
- 字段类型和必填规则；
- 外键引用校验；
- 重复 ID 检查；
- 未使用资源检查；
- 导出时间与数据版本；
- 可被测试脚本加载的明文源格式。

首章建立最小表集：

- `chapter`：章节目标和解锁；
- `map_node`：节点与连接；
- `event`：条件、选择和结果；
- `crew`：角色、职业、技能和特性；
- `ship_module`：船只模块；
- `enemy`：敌船与接舷单位；
- `reward`：战利品与用途；
- `dialogue`：剧情与航行对话。

旧表在未完成映射前保持只读，不能边重做边继续向旧表追加新体系。

### 10.4 存档与兼容

纵切阶段应明确区分：

- 旧版本存档：用于历史功能验证；
- V2 开发存档：用于首章重做；
- 测试存档：固定场景、战斗和节点入口。

在产品验证前不承诺旧存档完全迁移。若未来准备发行，再设计一次性迁移或
明确采用新存档。

## 11. 十六周迭代路线

### 阶段 0：设计与工程收口（第 1～2 周）

目标：建立唯一口径和稳定开发基线。

交付：

- 本文和系统取舍矩阵确认；
- 首章流程图与节点图；
- 首章数据表字段定义；
- 旧系统入口开关清单；
- 新存档和测试存档方案；
- 当前构建、首屏和旧主循环基线截图。

验收门槛：

- 产品、策划、程序和美术能用同一条流程解释首章；
- 每个首章功能有明确复用、重做或新建结论；
- iOS 模拟器与设备编译通过；
- 不再以互相冲突的旧 Excel 直接派发开发任务。

### 阶段 1：灰盒首章纵切（第 3～6 周）

目标：不追求最终美术，先完整跑通 20 分钟体验。

交付：

- 开场和皇家港最小流程；
- 一张小型迷雾海图；
- 四类船员和最小整备；
- 事件、敌船和返航节点；
- 舰炮战转接舷战；
- 首次符文线索与一次升级；
- 首章外系统隔离。

验收门槛：

- 新存档可无作弊完成首章；
- 没有必须进入商店或旧长期系统的步骤；
- 测试者不需要口头指导即可第一次出航；
- 所有失败路径可以返回港口或重新开始。

### 阶段 2：探索与战斗重做（第 7～10 周）

目标：让路线选择和双阶段战斗产生真实决策。

交付：

- 安全路线与风险路线；
- 航海士或情报对迷雾的影响；
- 一种舰炮结果向接舷传递的机制；
- 清晰的技能、状态和胜负反馈；
- 统一的首章数值模型；
- 可解释的撤退、失败和恢复。

验收门槛：

- 测试者能解释一次路线选择的利弊；
- 测试者能说出舰炮阶段如何影响接舷；
- 同一场战斗不存在关键结果无反馈；
- 数值调整通过源表完成，不直接散改 Lua 常量。

### 阶段 3：内容与美术定调（第 11～14 周）

目标：把灰盒纵切提升为能代表正式产品的样片。

交付：

- 四个英雄画面的正式构图与第一轮资源；
- 6～8 个首章事件；
- 首章剧情对白和航行对话；
- 一套敌船、接舷敌人和首领表现；
- 海浪、炮击、登船、胜利和诅咒音效；
- 首章图标、奖励和关键动画。

验收门槛：

- 缩略截图仍能辨认探索、舰炮、接舷和符文；
- 新画面在真机上明显优于旧版；
- 字号、触控、安全区和帧率通过真机检查；
- 正式资源没有使用临时生成占位图冒充成品。

### 阶段 4：测试与立项决策（第 15～16 周）

目标：判断是否值得进入前三海域和正式发行开发。

交付：

- 新手流程埋点或等价测试记录；
- 至少两轮外部目标用户测试；
- 首章问题分级和修复；
- 性能、崩溃、存档和资源检查；
- 下一阶段内容预算与商业模式建议。

立项门槛：

- 大多数测试者能独立完成首航；
- 大多数测试者能理解双阶段战斗；
- 返航后存在明确的第二次出航意愿；
- 测试者首先描述的是“海图、船战、符文或船员”，而不是“菜单和资源表”；
- 团队能够用现有流程稳定制作第二张地图、第二场战斗和新事件。

如果门槛未通过，应继续修正纵切，不直接扩建十六海域。
